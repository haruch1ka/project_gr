# 知識検証理論のまとめ

このアプリの「知識体系化」設計に援用するための、学問的な知識検証フレームワークの整理。

---

## 1. 科学的方法（Scientific Method）

```
観察 → 仮説設定 → 予測 → 実験 → 結果分析 → 理論化
                              ↑反証されたら仮説に戻る
```

### ポパーの反証主義（Falsificationism）

- 知識は「証明されるもの」ではなく「まだ反証されていないもの」として扱う
- 「白鳥はすべて白い」→ 黒い白鳥が1羽見つかれば即座に反証される
- どれだけ支持する事例を積み上げても「完全に証明」はできない

**→ アプリへの示唆：`verified` は「証明済み」ではなく「現時点で反証されていない」を意味する**

---

## 2. ベイズ推論（Bayesian Inference）

証拠が来るたびに確信度を連続的に更新するモデル。

```
事前確率（Prior）
  × 新しい証拠の尤度（Likelihood）
  = 事後確率（Posterior）→ 次のPriorになる
```

### 日常的な例

```
「今日雨が降る確率」

初期確信度（Prior）：「この季節、雨は30%」→ 30%
証拠①「天気予報が雨と言っている」→ 80%に更新
証拠②「外で傘を持っている人を見た」→ 90%に更新
```

### 重要な考え方

- 確信度は 0 か 100 ではなくグラデーション
- 証拠の質によって更新幅が変わる（信頼できる証拠は大きく動かす）

**→ アプリへの示唆：`confidenceScore` をベイズ的に設計する。支持経験でスコアが上昇し、反証経験でスコアが下降する。反証は支持より1.5倍重く扱う。**

## 3. エビデンスの階層（Evidence Hierarchy / EBM）

「どの情報をどれだけ信じるべきか」を整理した枠組み。医学の根拠に基づく医療（EBM）で発達。

```
高 ← メタ分析・システマティックレビュー（複数研究の統合）
     RCT（ランダム化比較試験：偶然を排除した実験）
     観察研究（自然な状態での観察）
     症例報告（個別事例の記録）
低 ← 専門家の意見・個人の経験
```

### 個人経験が低い理由

- **確証バイアス**：信じたい結果を見やすい
- **N=1**：1人の経験は統計的に一般化できない
- **プラセボ効果**：信じているから効いた気がする

### このアプリへの再解釈

このアプリは「自分の上達」という個人に閉じた問題を扱うため、エビデンス階層を再定義する必要がある。

| 情報源 | 重み（暫定） |
|--------|------------|
| Web上の複数一致した情報 | 低（仮説の種。Prior = 0.05 固定） |
| 自分の経験（1回） | 中 |
| 自分の経験（複数回再現） | 高 |
| 複数分野で共通して観測 | 最高（Meta Knowledge候補） |

### 尤度（Likelihood）とは

**「証拠がどれだけ役に立つか」のスコア。**

「毎日走ると持久力が上がる」という考えを確かめたいとする。

| 証拠 | 尤度 |
|---|---|
| 「5km走れるようになった」 | 高い（この考えが正しければ起こりそう） |
| 「変化なかった」 | 低い（この考えと矛盾する） |
| 「今日は晴れだった」 | ほぼゼロ（まったく関係ない） |

#### 証拠の数との違い

| | 何を決める |
|---|---|
| 尤度 | 1件の証拠で考えがどれだけ変わるか |
| 証拠の数 | 変化が何回起きるか |

証拠が多いと confidence が上がるのは「尤度が大きくなるから」ではなく、「小さな更新が何度も積み重なるから」。

#### このアプリでの扱い

数件の経験ログでは数式で計算できないため、Geminiが「この証拠は仮説にとって高・中・低どれか」を判断してスコアに変換する。

## 4. 各理論のアプリ設計への対応

| 学問的概念 | アプリへの対応 |
|---|---|
| 反証主義 | 反例経験がある限り `verified` にしない |
| ベイズ更新 | `confidenceScore` を証拠の積み重ねで連続更新 |
| エビデンス階層 | Web情報（低）< 経験1回（中）< 複数回再現（高）|
| 科学的方法のループ | フロー①〜⑤の理論的根拠 |

## 5. 多変数の同時変化問題（Confounding Variables Problem）

1回の経験ログには複数の条件が同時に含まれる（例：小潮 × 干潮 × 遠投 × 金色ワーム → 失敗）。どの変数が結果の原因かを帰属（attribution）できないという問題。

### 手法の調査結果

| 手法 | 適用可否 | 根拠 |
|------|---------|------|
| 因果推論（DoWhy・傾向スコア法等） | △ | 大量データが前提。N=40程度では doubly robust g-computation が最も安定するが、このアプリのログ量では適用困難 |
| DAG（有向非巡回グラフ） | ○ | ドメイン知識を事前構造として組み込める。データ不足でも「あり得ない因果経路」の除外が可能 |
| LLMによる因果グラフ推論 | ○ | 変数名のメタデータだけからLLMが因果グラフを推論できることが示されている（2024〜2025年の研究） |
| 実験計画法（DoE・直交表） | △ | ユーザーへの強制はNG。AIが「次に試す条件」をサジェストする形なら適用可能 |
| Contextual Bandit / 強化学習 | △ | ログ100件超から有効。コールドスタート問題があり初期は使えない |
| Grounded Theory（定性的研究） | ○ | 少ないデータからボトムアップでパターンを抽出する手法。LLMとの相性が最も良い。LOGOSプロジェクト（2025）でLLM駆動の自動化が実現されている |

### QuantifiedSelfドメインの知見

MITの**QuantifyMe**（単一事例実験計画：SCED）の重要な知見：「ユーザーが目標変数の範囲内に実際に行動できた日だけを分析対象にする」という絞り込みが交絡を大幅に減らす。少ないログでも適用可能。

### 小サンプルでの因果推論ベストプラクティス

1. **事前知識をPriorとして制約に組み込む**：ドメイン知識（「小潮は一般的に釣りにくい」等）をDAGやベイズ事前分布として組み込めば、データが少なくても推論が安定する
2. **ハード制約とソフト制約の使い分け**：「AがBに影響するのはあり得ない」を禁止エッジ（ハード）、「CがDに影響しやすい」を事前確率（ソフト）で表現
3. **帰属の不確実性を正直に出力する**：サンプルが不足している段階では「帰属できない」を正直に出力することがベストプラクティス。断言することは科学的誠実さに反する

## 6. 設計判断（解決済み・未解決）

- `verified` の廃止：`confidenceScore` の連続値のみで表現する（`status` フィールドは廃止済み）
- 反証の重み：**非対称採用（解決済み）**。反証は支持より1.5倍重く扱う。尤度ごとの係数は下表。

| 方向 | 尤度 high | 尤度 medium | 尤度 low |
|------|-----------|-------------|---------|
| 支持（supporting） | +0.10 | +0.06 | +0.02 |
| 反証（contradicting） | −0.15 | −0.09 | −0.03 |

- 個人経験のバイアス補正：どう設計に織り込むか（未解決）
