# 個人コーディングスタイル・作業規約

このファイルは Claude Code のグローバル設定として、すべてのプロジェクトに適用される個人の作業方針を定義する。

判断に迷う場面での行動原則は [DECISION_GUIDE.md](DECISION_GUIDE.md) を参照すること。

---

## 言語・コミュニケーション

- **応答は常に日本語**で行うこと
- コード内のコメントも日本語で書く
- 技術用語・識別子（変数名、関数名など）は英語のままでよい
- 応答は**簡潔に**。不要な長文の説明は省略する
- コードのみで意図が伝わる場合は説明を省略してよい
- 変更内容のサマリーを毎回末尾に追記しない

---

## コーディングの基本原則

### 最小原則

- 指示されたタスクの範囲内でのみ変更する
- 余分なリファクタ・抽象化・機能追加を勝手に行わない
- 似た処理が3箇所あっても、依頼がなければ共通化しない
- 新規ファイルの作成より既存ファイルの編集を優先する

### 既存スタイルへの忠実性

- コードベースに存在するスタイルに合わせること
- 命名規則・ファイル構成・型定義の書き方は既存パターンに従う

### コメントの方針

- **コメントは最小限**。自明なコードに説明コメントは不要
- 書くべきコメントは「なぜそうなっているか（WHY）」のみ
  - 隠れた制約、非自明な挙動の回避策、驚くような実装の理由
  - 外部制約（API 制限・バージョン制約・型定義不備）による代替実装の理由
- コンポーネントの Props interface・型定義には JSDoc 不要

## エラーハンドリング

- `null` は受け取った関数内で処理し、呼び出し元に伝播させない
- null チェックはガード節で行い、即 `throw` する（早期 `return` は void 関数のみ）
- 三項演算子内で `throw` が必要な場合は IIFE でラップする（`throw` は文であるため）
- エラーメッセージは日本語で記述し、失敗した対象を末尾に付記する

## 命名規則

| 対象                       | 規則       | 例                                |
| -------------------------- | ---------- | --------------------------------- |
| コンポーネント             | PascalCase | `UserCard`, `SubmitButton`        |
| 変数・関数                 | camelCase  | `formData`, `handleSubmit`        |
| 型・インターフェース       | PascalCase | `FormDataFormat`, `ImageMetadata` |
| CSS クラス                 | kebab-case | `card-header`, `embla__slide`     |
| ファイル（コンポーネント） | PascalCase | `UserCard.tsx`, `FormArea.tsx`    |
| ファイル（ユーティリティ） | camelCase  | `post.ts`, `validateForm.ts`      |

### 関数命名の接頭辞

- **`get*`**: データ取得・読み取り専用（`getDoc`, `getLayerIndex`, `getAllFiles`）
- **`set*`**: 状態変更・書き込み（`setColor`, `setDuration`）
- **`is*`**: 真偽判定（`isTargetFile`, `isFormatFile`）
- **動詞 + 対象**: それ以外のアクション（`handleSubmit`, `changeColor`, `removeItem`）

## ファイル・データ設計

- 静的コンテンツは**単一の真実の源**（Single Source of Truth）に集約する
- ページコンポーネントにはロジックを持たせない（ルーティングのみ）
- 再利用可能な UI は `ui/` ディレクトリに分離する
- 環境変数は `config/` や `.env` ファイルで管理する

### 責務分離の方針

- **読み取り専用**（データ取得）と**書き込み**（状態変更）はディレクトリまたはファイルで分離する
- 複数箇所で共有する処理は `util/` 等の共通層に分離する
- 実行環境が限定されるファイルはサフィックスで区別する（例: Node.js 専用なら `.node.ts`）

## Claude への指示

### 変更時の確認事項

- 破壊的な変更（ファイル削除・大きなリファクタ）は事前に確認を取る
- セキュリティリスクがある実装は警告してから進める
- 作業前後に `docs/TASKS.md` がある場合はそこを確認・更新する

### コードの実装方針

- 実装目的に応じてまずタスクを設定し、実現可能な粒度まで分解する
- 最小構成から一つずつ実装し、それぞれが正しく動作することを確認してから次へ進む
- 動作確認が取れていない状態で次のタスクに着手しない

### やってはいけないこと

- 依頼されていない機能の追加
- 依頼されていないコードのクリーンアップ
- 英語での回答・コメント（技術用語・識別子を除く）
