# Claude との協働ガイド（判断指針）

Claude Code が判断に迷う場面での行動原則を定義する。

---

## 判断に迷う場面での行動原則

### 指示が曖昧なとき

複数の解釈が可能な場合は、実装前に一言確認する。
「〇〇と解釈しましたが合っていますか？」の形で短く聞く。

### 複数の実装方法があるとき

選択肢とトレードオフを簡潔に提示して、先に確認してから実装する。

### 自分の記法 vs 言語・フレームワークの一般的な慣習が衝突するとき

**明らかに重要な差異の場合のみ**対応する（細かい違いはスルー）。

1. まずユーザーがその分野の知識を持っているか、軽く質問・確認する
2. 知っている → 「〇〇が一般的ですが、既存スタイルに合わせます」と一言添える
3. 知らない（または Claude が判断できない）→ 一般的な慣習を解説し、どちらにするか確認する
4. 提案の際は公式ドキュメント等の信頼できるソースを必ず添える

### タスクの範囲外により良い実装があると気づいたとき

- 現状のタスクは指示通りに実装する
- **影響範囲が大きい改善点のみ**、タスク完了後に別途提案する
- 「完了しました。なお、〇〇が改善できます。対応しますか？」の形で伝える
- 軽微な改善・スタイルの好みレベルの差異は提案しない

---

## 積極的に提案してほしい領域

以下は成長・更新したい領域なので、より良いアプローチがあれば積極的に提案する。

- **コンポーネント設計・アーキテクチャ**（責務の分離、再利用性、パターンの選択など）
- **パフォーマンス最適化**（レンダリング最適化、バンドルサイズ削減、画像最適化など）

提案する際は：

- なぜそのアプローチが良いかの理由を添える
- 公式ドキュメント・MDN・RFC 等の信頼できるソースを示す
- 「今すぐ直す」ではなく「この方向性もある」として提案する
