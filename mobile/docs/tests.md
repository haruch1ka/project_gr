# テスト解説

`pnpm test` で実行される自動テストの一覧と意図をまとめる。

---

## 実行方法

```bash
pnpm test               # 全テスト（オフライン・高速）
pnpm check:prompts      # 実Gemini APIを叩くプロンプト確認スクリプト（手動実行）
```

---

## ファイル構成

```
src/
  services/__tests__/gemini.test.ts   # Geminiサービス層のロジック
  utils/__tests__/knowledge.test.ts   # 知識の色・ラベル変換ユーティリティ
  utils/__tests__/chat.test.ts        # チャット用プロンプトビルダー
scripts/
  check-prompts.mjs                   # プロンプト実動確認スクリプト（jest外）
```

## gemini.test.ts

### SCORE_DELTA

`updateKnowledgeFromExperience` が経験ログに基づいて確信度スコアを更新するときの増減値テーブル。

| テスト | 意図 |
|--------|------|
| `supporting/highが最大増加量` | 証拠強度が high > medium > low の順に増加量が大きいことを保証 |
| `contradicting/highが最大減少量` | 反証も同様に high ほど大きく減少することを保証 |
| `neutralはすべて0` | 関係なしの経験はスコアを動かさないことを保証 |
| `supportingとcontradictingは対称` | 支持と反証が同じ絶対値で逆方向に働くことを保証 |

> なぜテストするか：スコア更新は知識の「確信→仮説→疑問」遷移を決定するため、増減の大小関係が崩れると経験が正しく反映されなくなる。

---

### isValidCategory

Geminiが生成したカテゴリ名が「単一概念の名詞1語」ルールを満たしているかを判定する関数。

**有効なケース**

| 入力 | 理由 |
|------|------|
| `ルアーカラー` | 複合語だが単一概念 |
| `キャスト` | 単一名詞 |
| `回復・睡眠` | `・` による同類並列は許容 |

**無効なケース**

| 入力 | 検出理由 |
|------|---------|
| `木曽川ルアーカラー攻略` | 11文字超（長すぎる） |
| `光量とルアーカラー` | 助詞「と」で2概念を結合 |
| `赤金やチャート` | 助詞「や」で2概念を結合 |
| `（空文字）` | 空文字 |

> なぜテストするか：Geminiのプロンプト修正後も複合名称が生成された場合、`generateHypotheses` がリトライを発火するかどうかの判断をこの関数に委ねている。バリデーションルールの破損を早期検知するため。

---

### reorganizeIntoFolders

Geminiを使って知識アイテムをフォルダ構造に割り当てる関数。外部APIを `fetch` のモックで差し替えてテスト。

| テスト | 意図 |
|--------|------|
| `知識が空なら空配列を返す` | 0件のとき Gemini API を呼ばないことを確認（無駄なリクエスト防止） |
| `_idのない知識は除外されて空配列になる` | DB未保存のアイテムが誤ってプロンプトに含まれないことを確認 |
| `Geminiが返したアサインをそのまま返す` | APIレスポンスがそのまま呼び出し元に渡されることを確認 |
| `既存フォルダ名がプロンプトに含まれる` | 既存フォルダを参考情報として渡しているかをリクエストBodyから検証 |
| `currentFolderがプロンプトに含まれる` | `folderId → フォルダ名` の解決結果がプロンプトに入るかを検証 |

> なぜテストするか：プロンプトの構築ロジックが変わると再整理の品質が無言で劣化する。fetchをモックすることでAPIコストなしに構造を検証できる。

## knowledge.test.ts

`knowledgeColor` と `knowledgeLabel` のテスト。知識アイテムの `type` と `confidenceScore` から表示用の色・ラベルを返すユーティリティ。

### knowledgeColor

| スコア / タイプ | 期待カラー | 意味 |
|----------------|-----------|------|
| distilled（任意スコア） | `colors.primary` | 経験から直接得た発見は常に最高表示 |
| hypothesis ≥ 0.7 | `colors.primary` | 高確信 |
| hypothesis 0.3〜0.69 | `colors.textSecondary` | 通常仮説 |
| hypothesis < 0.3 | `colors.danger` | 疑問・要検証 |

| テスト | 意図 |
|--------|------|
| `distilledは常にprimaryカラー` | スコアに関係なく発見扱いになることを保証 |
| `score >= 0.7 → primary` | 高確信の閾値（0.7）が正しく機能することを保証 |
| `score 0.3〜0.69 → textSecondary` | 中間域の境界値（0.3・0.69）を網羅 |
| `score < 0.3 → danger` | 低確信の赤表示が閾値どおりに出ることを保証 |

---

### knowledgeLabel

`knowledgeColor` と同じ閾値で「発見」「確信」「仮説」「疑問」のラベルを返す。

| テスト | 意図 |
|--------|------|
| `distilledは常に「発見」` | 経験析出型の一貫性を保証 |
| `score >= 0.7 → 「確信」` | 高確信ラベルの閾値を確認 |
| `score 0.3〜0.69 → 「仮説」` | 中間ラベルの境界値を確認 |
| `score < 0.3 → 「疑問」` | 低確信ラベルの境界値を確認 |
| `colorとlabelの一貫性` | primary色 → 「発見」か「確信」のいずれかになることを横断検証 |

> なぜテストするか：UI上の色とラベルが食い違うと（赤なのに「確信」など）ユーザーに誤情報を与える。閾値を1か所変更したときに両方のテストが落ちることで整合性の崩れを検知できる。

## chat.test.ts

`storageKey` と `buildSystemPrompt` のテスト。チャット画面でGeminiに渡すシステムプロンプトの生成ロジック。

### storageKey

| テスト | 意図 |
|--------|------|
| `フィールド名からキーを生成する` | `chat_history:<field>` 形式になることを保証 |
| `空文字列でも動作する` | フィールド未設定時にクラッシュしないことを確認 |

---

### buildSystemPrompt

| テスト | 意図 |
|--------|------|
| `フィールド名がプロンプトに含まれる` | Geminiに分野コンテキストが渡されることを確認 |
| `知識なし・経験なしのフォールバック文言` | 空の状態でもプロンプトが壊れないことを確認 |
| `hypothesis・低スコアが「仮説」ラベル` | 確信度50%の知識が `[仮説 50%]` 形式で入ることを確認 |
| `hypothesis・高スコアが「確信」ラベル` | 確信度85%が `[確信 85%]` 形式で入ることを確認 |
| `distilledが「発見」ラベル` | 経験析出型が `[発見 60%]` 形式で入ることを確認 |
| `低スコア（0.3未満）が「疑問」ラベル` | 疑問レベルの知識が `[疑問 10%]` 形式で入ることを確認 |
| `経験ログが日付・メモ形式で含まれる` | 経験ログが `YYYY-MM-DD: メモ` 形式で挿入されることを確認 |
| `経験ログは最大5件` | 多数の経験がある場合に古いものが省かれることを確認（プロンプト肥大化防止） |
| `信頼度スコアは四捨五入パーセント` | `0.754 → 75%` の丸め処理が正しいことを確認 |

> なぜテストするか：システムプロンプトの品質がGeminiの回答品質に直結する。プロンプトの構造が変わった際に意図しない情報の欠落・重複を即座に検知するため。

## check-prompts.mjs（jest外）

`pnpm check:prompts` で手動実行するスクリプト。実Gemini APIを呼び出して、プロンプト修正の効果を確認する。

| チェック項目 | 内容 |
|-------------|------|
| `generateHypotheses` カテゴリ名 | 4ケース（釣り×2・筋トレ・読書）で `isValidCategory` を通過するか |
| `reorganizeIntoFolders` フォルダ名 | 5件の釣り知識を整理したときのフォルダ名が単一概念になるか |

jest に含めていない理由：実APIを呼ぶためコストがかかり非決定的。プロンプト変更時に手動で確認する用途に限定している。
